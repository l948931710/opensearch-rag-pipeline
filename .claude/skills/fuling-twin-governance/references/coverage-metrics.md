# 内容包含性度量 — 清洗规则、双门判定与三个实测假象

脚本：`scratch/twin_content_coverage.py`。目的：回答"keep 是否包含 retire 的全部
内容"（cov(r⊆k)）与"若翻转会丢什么"（cov(k⊆r)），作为退役/翻转的决定性证据。

## 为什么必须只比"净源文本"

chunk_text 是**复合体**：源文本 + VLM/OCR 增强段 + 嵌入前缀。直接整串对比时，
增强段的随机差异（VLM 描述措辞、OCR 噪声）会淹没真实的源内容差异，结论完全失真。
本项目踩实过三个假象（每个都曾让一轮全量对比作废）：

1. **`image` 类型 chunk 是 VLM 产物**（"【文档:X】图片描述: …"），不是源文本。
   连同 `procedure_parent`（步骤卡汇编，双计）与 `visual_knowledge` 一起排除。
2. **增强段是多行段**：`[图片内容]/[补充图示]/[图片OCR]/图中①…` 标记后面跟多行。
   按行剥前缀会漏掉后续行 → 必须**从首个标记截断到 chunk 尾**（chunker 总把增强
   段追加在源步骤文本之后，截断安全）。
3. **所有 chunk_text 带 `【文档:标题】` 嵌入前缀**（pipeline_nodes 写库时加），
   且标题含 `.docx/.pdf` 扩展名——两侧必然不同。要**整体剥除该 token**，绝不能把
   它当截断标记（曾把全部文本截成 0 字，所有组假 100% 包含）。

## 双门判定

- **句级精确**（主）：归一化（仅留 CJK+字母数字，小写）→ 按 `。；;！？\n` 切单元
  （≥6 字、去重）→ A 的单元是否为 B 全文子串，按**字符长度加权**。
- **8-gram 模糊**（副）：n-gram 集合包含率。对表格单元格顺序/合并单元格渲染差异
  （"仓管仓管"vs"仓管"、"200"vs"2001"）鲁棒。
- 包含成立 = `cov≥97% 或 fz≥95%`。两门接近 → 真差异；精确低而模糊高 → 表格渲染
  变体居多。

## 判读注意

- **长度加权会放大单条长串**：一条几百字的表单数据串缺失能把 cov 压到 65%，而
  按单元数是 38/40。报告时两个口径都看，并把**全部缺失单元逐条列出定性**——
  裁决依据是缺的"是什么"，不是百分比本身。
- **图无法用文本证明包含**：图维度只用 OCR 文本近似 + 图数对比，单独列报；
  retire 图多于 keep 的组必须走图审计目检（见 SKILL.md ③）。
- **coverage 不随回灌改变**：差距源自源文件与抽取器（pdf 表格抽取丢行、docx
  文本框丢失、扫描件无文本层），重切只改图绑定。若想靠回灌改善 coverage，先去
  修抽取器（参照 docx `_textbox_texts` 缺口的修法），再回灌再复算。
- 复算是幂等的：翻转 keep 后重跑脚本即可复验新方向的包含性。
