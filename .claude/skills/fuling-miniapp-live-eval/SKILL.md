# 富岭小程序 LIVE 实测 + 答案诊断溯源

两条相互衔接的工作流：**A. 实测验收**（在真实 UI 上验证回答质量）发现问题后进入
**B. 诊断溯源**（四层判定树定位根因）。先读 `references/environment.md` 拿到全部
端口/命令/选择器等硬事实；诊断时按症状路由到 `references/diagnosis-trees.md` 的
对应判定树；动手前扫一遍 `references/pitfalls.md`（每条都是实战踩过的）。

## 公平性四原则（一切实测的前提）

实测结论要能代表生产，必须同时满足，缺一条结论就不可信：

1. **配置同构**：本地 serving 用与 SAE 生产完全一致的环境形态（rerank ON、守卫 ON、
   上下文预算等，精确命令见 environment.md）。改过任何 serving 代码后**必须重启
   uvicorn**——Python 进程不会热加载，旧逻辑会让你验证一个不存在的版本。
2. **语料同面**：`RAG_ENV=prod_ro` 只读直连生产 HA3/RDS——检索面与线上一字不差，
   新入库文档即测即见。
3. **题集延续**：优先复用既有题集（`scratch/local_e2e_answers.json` 25 题、
   金集 `eval_harness/goldset/golden_full.json`），新增题型补充而不是替换——
   这样结果能与历史报告对照。
4. **渲染真值**：图片必须验到浏览器网络层（签名 URL 返回 200）与 DOM 层
   （`naturalWidth>0`）——API 返回 blocks 不等于用户真的看到了图。

## 工作流 A：小程序 LIVE 实测验收

1. **起后端**（Bash 后台，命令与环境变量逐字见 environment.md §后端）。
2. **起前端**：`preview_start` 配置名 `miniapp-prototype`（:4599 静态服务）。
3. **接 LIVE 桥**：导航到 `/?api=http://127.0.0.1:8001`——标题栏出现「LIVE实测」
   即桥接成功；没有它说明走的是脚本化 mock 对话，测了也白测。
4. **选代表题**（按验证目标组合，五类题型）：

   | 题型 | 验证什么 | 例 |
   |---|---|---|
   | 回灌/图文步骤卡题 | 图文绑定、步骤样式、图方向 | 吸塑扫码报检怎么操作 |
   | 大手册深层小节题 | 扩展防洪、深层检索、拒答恢复 | 改员工档案手机号怎么弄 |
   | 新入库语料题 | 新文档即刻可检索、跨文档融合 | （选最近注册的文档提问） |
   | 库外负样本 | 未找到卡片态、不凑无关来源 | 食堂今天有什么菜 |
   | 用户报障同款题 | 复现用户截图里的现象 | 按用户原话提问 |

5. **逐题取证**：`preview_fill` `#input` → `preview_click` `#send` → 时序 gate
   （生成约 6-18s + 打字机渲染，gate 写法见 pitfalls §sleep 拦截）→ DOM 取证
   （现成 eval JS 模板见 environment.md §DOM 取证）。每题记录：步骤数、图数与
   `loaded` 状态、captions/alt、是否拒答、来源面板构成；关键题截图留证。
6. **判读**：通过 → 下一题；异常 → 进入工作流 B。**先别改任何东西**。
7. **收尾**：`pkill -f "uvicorn opensearch_pipeline.api:app"` 停后端；preview 服务
   留给用户继续玩或一并停掉；结论与工件路径写进答复（截图、json、报告归档惯例
   见 environment.md §产物归档）。

## 工作流 B：四层诊断溯源

答案质量问题的根因必落在四层之一。**自上而下逐层取证，每层有专用探针，
拿到证据再下结论**——今天修过的每个 bug 都曾被"想当然"指错过层。

| 层 | 管的事 | 探针 |
|---|---|---|
| L1 数据层 | chunk 是否存在/绑定是否正确/版本对照/文档孪生 | RDS 只读查询（判定树内附 SQL） |
| L2 检索层 | 进没进池/排名/rerank 分/扩展构成/进没进 context | `scripts/diag_answer_chain.py` |
| L3 生成层 | 提示形态/LLM 是否引用/拒答判定/引用泄漏 | `scripts/probe_citation_x3.py` |
| L4 渲染层 | 配额分配/近重抑制/签名 URL/DOM 实载 | 合成引用隔离 + preview 网络/DOM |

**按症状路由**（判定树全文在 `references/diagnosis-trees.md`）：

- 图缺失/图不对/图记得有却没出来 → §图缺失判定树（L1→L4 八步）
- 拒答/未找到（但你认为该答出） → §拒答判定树
- 图或引用重复/来源出现两次 → §重复判定树（含孪生文档检测 SQL）
- 答案内容错/漏要点 → 先跑 L2 探针看目标小节进没进 context，再走拒答树后半段

**铁律**：
- 生产访问一律经 `opensearch_pipeline.prod_access`（只读默认），严禁手工解析
  `.env.production`；生产写（退役/重置）必须用户逐次授权 + 当日 `PROD-RW:<日期>` 令牌。
- **偶发与确定要分开**：LLM 引用/拒答是概率行为，单次异常先 ×3 重复
  （`probe_citation_x3.py`）或 API 直连复测再定性；"UI 一次拒答"不构成回归结论。
- 结论里写清：根因层、证据（探针输出）、是否可复现、修复建议与验证方式。

## 批量复测

修复后做题集级回归用 `scripts/collect_questions.py`（题集 json → 逐题
rank/图数/拒答/延迟 json，自动对照期望文档）。与历史轮次对比时注意标题别名
漂移（pitfalls §标题匹配假阴）。

## 诚实报告规范

实测报告必须包含：环境形态声明（四原则逐条）、每题原始数据（非汇总形容词）、
异常的复现性定性、未覆盖面。曾因少声明"serving 未重启"浪费过一轮验证——
环境形态就是结论的一部分。
