# HA3 RAG — End-to-End Evaluation Harness

A comprehensive, **live, read-only** evaluation of the rebuilt HA3 index (`fuling_kb_chunks`)
and the full serving stack. Reuses the existing eval assets (`validate_v2`, `diag_*`,
`mm_answer_metrics`/`mm_judge`, the official 200-case xlsx, and the 51-query JSON GT) and
adds the missing piece: one orchestrated pipeline that produces a single PASS/FAIL report
across index health → retrieval → calibration → answer quality → multimodal → permission →
latency.

## Fairness design (why you can trust the numbers)

- **Judge ≠ generator.** Answers are generated by Qwen (`qwen3.6-plus`); they are judged by an
  **independent Claude panel** (Qwen judging itself = self-evaluation bias). The panel is
  blinded to model identity, anchored to a 1–5 rubric, and grounded in the retrieved context
  + gold answer points. Inter-judge agreement is reported.
- **Thinking mode OFF.** Generation forces `enable_thinking=false` and verifies no
  `reasoning_content` leaked (`reasoning_leak_count` must be 0), so answer quality reflects the
  intended production behavior.
- **No unfair recall charges.** A positive is scored for retrieval only if its gold document
  actually exists in the live index (resolved via `document_meta`). Unresolved golds (e.g. the
  finance `CWD-*` docs not in this rebuild) are **excluded and reported**, never counted as misses.
- **Permission-correct exclusions don't count as failures.** A gold doc that is `dept_internal`
  is not publicly retrievable; those cases are split out, not charged against public recall.
- **Honest stats.** Bootstrap 95% CIs on every aggregate (n≈50 is small), negatives/refusals/
  errors all reported, strict + lenient title matching, no silent truncation.
- **Read-only.** Every HA3/RDS call is read-only. No writes, no deletes.

## Layers

| Layer | What it proves | Key gates |
|---|---|---|
| **L0** index health | rebuild is healthy; the `order=DESC` InnerProduct trap isn't biting | status IN_USE; docCount≈RDS active (loss=fail); dense self-query rank-1@~1.0; sparse built; stored≈fresh vector (cos~1.0) |
| **L1** retrieval ranking | the index finds the right doc | recall@1/3/5/10, MRR, nDCG@10 (+CIs), per dept/difficulty; negative over-confidence probe; latency |
| **L2** score calibration | the 高/中/低 labels (8.0/5.0) still fit | fraction of correct top-1 hits reaching 高/中; pos-vs-neg score separation |
| **L3** answer quality | answers are faithful & don't over-refuse | deterministic (refusal, source-leak, marker-leak, reasoning-leak) + **Claude panel** (faithfulness/correctness/completeness/relevance, fabrication) |
| **L4** multimodal | image-text interleaving | marker validity, orphan/dangling, availability (reuses `mm_answer_metrics`) |
| **L5** permission | dept filtering + injection safety | gated docs hidden on public path, visible to authorized dept, injection-safe |
| **L6** latency | serving speed | p50/p95/p99 of retrieval (folded into L1) and generation (L3) |

## Gold set

Built by `goldset/build_goldset.py` from **both** sources the user asked for:
- the official 200-case `一期模型评测集和评测模板_v2.0(1).xlsx` (RAG retrieval + NLQ + source-attribution, with the 85%/95% rubric), and
- the 51-query `text_eval_queries.json` (difficulty-stratified, keyword GT) + 19 multimodal queries.

Each case is resolved against the live inventory; `golden_50.json` is a stratified
representative subset (modules × depts × difficulty × kind, ~24% negatives, image queries
included). `golden_full.json` is the full union for opt-in larger runs. See
`goldset/resolution_report.json` for coverage (incl. which golds are not in the index).

## Running

```bash
# 0) (re)build the gold set against the live inventory (read-only RDS)
python -m eval_harness.goldset.build_goldset

# 1) run all layers live (read-only); writes reports/run_<ts>/{report.md,report.json,judge_bundle.json}
python -m eval_harness.run_eval run

#    quick smoke / subset
python -m eval_harness.run_eval run --limit 5 --layers l0,l1,l2

# 2) judge the bundle with an INDEPENDENT Claude panel -> judge_verdicts.json
#    (run the panel via the project's multi-agent workflow over reports/run_<ts>/judge_bundle.json;
#     verdicts file shape: {"panels":[{"judge":"claude-judge-1","verdicts":[<VERDICT_ITEM_SCHEMA>...]}, ...]})

# 3) merge judge verdicts into the FINAL report
python -m eval_harness.run_eval merge \
  --results reports/run_<ts>/report.json \
  --verdicts reports/run_<ts>/judge_verdicts.json
```

`enable_thinking=false`, the public HA3 endpoint, live table, and non-simulate mode are forced
by `envboot.py` on import — no shell setup needed beyond the existing `.env` / `.env.production`.
