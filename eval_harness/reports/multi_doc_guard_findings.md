# 跨文档检索 + 低置信度护栏 — 配对 live A/B 评测结论

**日期:** 2026-06-09 · **索引:** `fuling_kb_chunks`（live，只读）· **配置:** rerank ON（与生产一致）、top_k=7
**数据:** `multi_doc_ab_v2.json`（权威，含全部修复）；v1 = `multi_doc_ab.json`（方向一致的复现）
**方法:** 同 case 配对多臂（exact McNemar + 配对 bootstrap CI + 非劣界 −5pp）；瞬时故障重试 + 出错整对剔除（v2 实际 0 错误）；预热轮平衡冷启动；触发记录器隔离各臂。
**被测机制（全部默认关闭落库）:** ① `RAG_LOW_CONFIDENCE_GUARD` 低置信度护栏（generation 侧软闸门）② `RAG_MULTI_QUERY_MODE` 多意图查询分解 ③ `RAG_DOC_DIVERSITY_CAP` 文档多样性限额。

## 结论一览

| 机制 | 核心证据（v2） | 成本 | 裁决 |
|---|---|---|---|
| **低置信度护栏** | 负例拦截 **0.50 → 0.654**（+4 翻转 / 0 反向，McNemar p=0.125）；正例误拒 **0/50（两臂均 0）**；触发正例的关键词覆盖 0.917 不变。v1 完全复现（含锚定版 hard_refusal 重打分）。 | 0 延迟（只加一段 system prompt） | ✅ **建议生产开启** |
| **查询分解（auto）** | per-doc coverage +1.0pp（CI [0, +3.1]）；full-coverage 0 翻转；收益集中在可分解意图（QA-100：1→3/4 docs）；单文档 50 题 0 回归（recall@1 0 翻转）；负例触发率 0.04。 | ~30% 查询触发判别调用，多文档集均延迟 +0.76s | ⏸️ **代码保留、默认 off**（收益不抵延迟） |

## 为什么"多文档 R@1≈8%"修不动 —— 缺口的真实构成

24 个跨文档 gold case 配对诊断（见 v2 `multidoc.strata` + 逐案 doc_titles）：

1. **枚举型 gold（10/24，预期 4~6 份文档）**：如 QA-106"入职到离职全生命周期涉及哪些制度"预期 6 份制度。top_k=7 的上下文**结构上装不下**；且通用子查询（"在职管理制度"）点不出 gold 要求的具体方面（考勤/休假/培训）。这是任务定义问题（"列出所有相关文档"≠RAG 问答），不是排序问题。该层 cov_frac 仅 0.38，分解仅 +2.5pp。
2. **可分解多意图（≈7/24 触发）**：分解真实有效——QA-100"女职工和未成年工权益"从 1/4 → 3/4 份目标文档（两臂复现）。但这类 case 太少，拉不动总体。
3. **近重复文档家族**：QA-91 的 top-7 被《外来人员入厂告知书（新）》《（松门）》两个**不同 doc_id** 的变体占满——文档级限额对这种家族失效（这也是 cap=4 无增益的根因之一；RDS 里 249 份已转换旧注册行属同类问题，见 coverage_gap_findings.md）。
4. **2~3 文档层（14/24）**：full-coverage 0.357，三种机制全都没动它——第二份文档可检索但排不进 top-7，且问题单意图无从分解。要动它只能加大上下文预算（top_k/max_context_chars 重新权衡）或语料侧合并近重复文档。

## 护栏设计依据（为什么不是硬闸门）

离线标定（`gate_calibration.py`，251 题路由重排 top-1 分重放）：正/负分布重叠严重——拦截 >30% 负例的任何阈值都要误拒 ≥20% 正例；部分陷阱负例（问不存在的型号/编号，库里有近似文档）分高达 0.93+。曾公布的 Youden J 0.61/0.66 是**只在 rank==1 正确命中上**算的（标签标定视角），不是闸门工作点。故弃硬闸门，改为低置信带（top rerank 分 < 0.8）注入"逐条核对、不对题必须拒答"指令，由读得到内容的 LLM 做二级判别：

- 拦截新增的 4 例全是该拦的：如"ERP里库存还有多少？"原回答跑题讲"如何查库存"，护栏后干净拒答。
- 护栏未触发而仍作答的 9 个"负例"top 分全 >0.8，其中多例（车辆停放图、饭卡通知、文档编号）答案内容看起来真实存在于索引——**疑似 gold 标注噪声**，本就不该拦。

## 补充评测（同日）：检索预算 k7 vs k10 + 上下文容量实测（`budget_ab.py`）

配对（24 跨文档 + 50 单文档，mode=off cap=0，预热轮，0 错误）：

| 对比 | full_coverage | cov_frac Δ | 枚举层（≥4 docs）cov_frac | 单文档回归 |
|---|---|---|---|---|
| k7 → k10 | 0.208 → **0.250**（+1 翻转/0 失）| +2.1pp（CI [0, +5.2]）| 0.382 → **0.432**（+5pp）| **0 翻转/50**，非劣 ✓ |

**更重要的发现——现行配置已在截断**：拼接后上下文实测（字符）：

| 段 | k7 p50 | k7 p90 | 超 6000 率 | k10 p50 | k10 超 8000 率 |
|---|---|---|---|---|---|
| 跨文档 | 6,702 | 8,471 | **75%** | 9,248 | 88% |
| 单文档 | 6,386 | 9,289 | **60%** | 9,318 | 70% |

即 `RAG_MAX_CONTEXT_CHARS=6000` 对**今天的 k7+stitch±1 生产配置**就已偏小：六成以上的问答在
`_format_context` 处把已检回的相关内容截掉（通常截尾 chunk，p90 场景截掉 ≥2.5k 字符）。
6000 这个值锁定于 stitch 膨胀未计入时（"top_k=7 + stitch ≈ 5.7k"的旧估算）。

**建议:**
1. （低风险，先行）`RAG_MAX_CONTEXT_CHARS` 6000 → **10000**：k7 下 p90 也不再截断；
   代价是 prompt 变长（生成延迟/费用小幅上升）。
2. （待证）top_k=10 的检索增益真实但小（+1 例 full-coverage，枚举层 +5pp），且需 ~13k 上下文
   才不截断——**先做生成侧评审团 A/B**（k7/10k vs k10/13k）确认答案质量净增益再动 top_k。
